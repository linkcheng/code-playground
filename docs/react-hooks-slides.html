Props & State — 核心概念

PropsState
本质外部传入的数据（只读）组件内部管理的数据（可变）
代码<Comp label="hi" />const [n, setN] = useState(0)
修改不能修改，由父组件控制通过 setState 修改

状态封装

组件用 useState 自管理，独立可复用，但父组件无法直接控制

状态提升

状态放父组件，通过 props 传入子组件，父组件拥有完全控制权

选择依据

谁需要这个数据，状态就放在谁那里

Hooks 调用规则 — 为什么必须顶层调用

React 用链表 + 位置索引匹配 Hook，不是靠名字或 key

渲染 #1: useState(0) -> useEffect() -> useState('') | | | Hook 链表: [hook#0] -> [hook#1] -> [hook#2] 渲染 #2: useState(0) -> useEffect() -> useState('') | | | 复用 hook#0 复用 hook#1 复用 hook#2

不能放在 if/else、for/while、嵌套函数

条件渲染导致 Hook 数量不一致 -> 索引错位 -> 状态混乱

始终在组件顶层、按固定顺序调用

自定义 Hook 内部的 Hook 也挂载到宿主组件链表上

useState — 组件内状态

const [count, setCount] = useState(0) // setCount(1) -> 触发重新渲染 -> 新值用于生成 JSX

三个核心陷阱

不可变性

user.name = 'Jerry'; setUser(user) setUser({...user, name: 'Jerry'})

React 用 Object.is() 比较引用

setState 是替换

setUser({ age: 25 }) // name 丢失! setUser(prev => ({...prev, age: 25}))

函数式更新

setNum(num + 1) // 闭包旧值 setNum(prev => prev + 1) // 最新值

useState — 进阶技巧

惰性初始化

// ❌ 每次渲染都执行 const [data] = useState(expensiveCalc()) // ✅ 只在首次渲染时执行 const [data] = useState(() => expensiveCalc())

函数式更新的参数来源

setCount(c => c + 1) // c 由 React 内部传入，保证最新 // React: 暂存函数 -> 取最新 state // -> 调用 -> 替换旧状态 // 两种调用方式： setCount(5) // 直接传值 setCount(c => c + 1) // 传函数

注意：useState(fn) 把 fn 的返回值当初始值；setCount(fn) 把 fn 当更新函数。同样传函数，语义完全不同。

useReducer — 复杂状态管理

用户操作-> dispatch(action)-> reducer(state, action)-> 新 state-> UI 更新
const reducer = (state, action) => { switch (action.type) { case "changeName": return {...state, name: action.payload} case "changeAge": return {...state, age: Number(action.payload)} default: return state } } const [info, dispatch] = useReducer(reducer, initialState) dispatch({ type: "changeName", payload: "新名字" })

Reducer 必须是纯函数：不直接修改 state，用 {...state, key: newVal} 返回新对象

useContext — 跨层级传递数据

createContext-> Provider 注入-> useContext 消费
// 1. 创建 export const ThemeContext = createContext({ theme: "light" }) // 2. 提供 <ThemeContext.Provider value={{ theme, toggleTheme }}> <Child /> </ThemeContext.Provider> // 3. 消费（推荐封装自定义 Hook） const { theme } = useContext(ThemeContext)

最佳实践：封装 useTheme()，消费方不用知道用哪个 Context。没有 Provider 时拿到 createContext 的默认值。

useEffect — 副作用 Hook

执行时机：渲染完成、浏览器绘制画面之后执行

每次渲染后

useEffect(() => { ... })

只在挂载时

useEffect(() => { ... }, [])

依赖变化时

useEffect(() => { ... }, [count])

依赖变化机制

React 用 Object.is() 对比依赖项前后值，不同则重新执行 effect。

useEffect — 清理函数

渲染1后 -> 执行函数体 -> 清理函数存起来 渲染2后 -> 先执行上次清理 -> 执行新函数体 -> 新清理存起来 ... 卸载时 -> 执行最后一次清理
useEffect(() => { const ws = new WebSocket(url) return () => ws.close() // 清理函数，被 React 存起来 }, [url])

清理函数不是定义时执行，而是被React 保管

在下次 effect 执行前或组件卸载时调用

useMemo & useCallback — 缓存

useMemouseCallback
缓存的是计算结果（值）函数引用
写法useMemo(() => val, deps)useCallback(() => fn, deps)
等价useCallback(fn, deps) === useMemo(() => fn, deps)
场景用不用
不传子组件、不放进依赖数组不用
函数传给 React.memo 子组件useCallback
对象/数组传给子组件useMemo
函数放进 useEffect 依赖useCallback

核心问题：引用相等 — 每次渲染 () => {} 都是新函数、{} 都是新对象。

React.memo 协作 — 三位一体

父组件重渲染 -> 所有子组件无条件跟着重渲染（默认行为）

React.memo

浅比较 props，相同则跳过渲染

const MC = React.memo(Child)

useCallback

稳定函数 props 引用

const fn = useCallback(() => {...}, [])

useMemo

稳定对象 props 引用

const s = useMemo(() => ({...}), [])

三者缺一不可：只用 useCallback 不用 memo -> 子照样渲染；只用 memo 不用 useCallback -> 引用变化 memo 失效。原始类型 props 天然稳定，只需要 memo。

useRef — 跨渲染的可变引用

useRefuseState
修改后不触发渲染触发渲染
场景DOM 引用、定时器 ID、标记位需要显示在界面上的数据

对象 ref

const ref = useRef<HTMLInputElement>(null) <input ref={ref} /> // 挂载时: ref.current = DOM // 卸载时: ref.current = null

Callback ref

<div ref={(node) => { if (node) { setHeight(node.getBoundingClientRect().height) } }} /> // 感知 DOM 绑定时机

Hooks 执行阶段 — 全景时序

1. 触发更新: setState / dispatch / context 变化 | 2. Render 阶段 -- 执行组件函数，计算 UI（纯计算，可中断） useState / useReducer / useContext / useMemo / useCallback / useRef | 3. Commit 阶段 -- 同步更新真实 DOM（不可中断） useInsertionEffect -> DOM mutation -> Callback ref -> useLayoutEffect | 4. 浏览器绘制 Paint | 5. useEffect -- 绘制后异步执行（请求、订阅、定时器）

一句话：Render 算 UI -> Commit 改 DOM -> LayoutEffect 绘制前处理 -> Effect 绘制后处理

Commit 阶段 — 三个同步方法

useInsertionEffectCallback refuseLayoutEffect
顺序DOM 前DOM 写入时DOM 后、绘制前
读 DOM?否是（当前节点）是（所有更新）
阻塞绘制?是是是
场景CSS-in-JS 注入测量、初始化布局修正、防闪烁
频率极低中等偶尔
insertionEffect -> React 写入 DOM -> callback ref(node) -> useLayoutEffect -> Paint -> useEffect

经验法则：99% 场景用 useEffect。需要同步读取/修改 DOM 布局才用 useLayoutEffect。

flushSync — 强制同步刷新

React 18 默认批量处理 state 更新，flushSync 打破批量机制

默认批量更新

setState(A) -> setState(B) -> setState(C) | 一次性渲染

flushSync 强制刷新

flushSync(() => setState(A)) | 立即渲染，DOM 已更新 setState(B) -> 留到下一批
import { flushSync } from 'react-dom' flushSync(() => { setSelectedId(id) }) // DOM 已更新，可同步读取 document.getElementById(`item-${id}`)?.scrollIntoView()

99% 场景不需要。主要与第三方库（动画、滚动、测量）DOM 互操作时使用。

命令式 vs 声明式

"谁负责同步 DOM"是 React 开发中的核心决策

维度命令式 Imperative声明式 Declarative
DOM 操作事件处理器直接调用useEffect 统一同步
同步路径分散（事件 + DOM 事件兜底）集中（Effect 一条路）
组件模式非受控（状态在组件内）受控（prop 驱动）
可复用性低（状态与逻辑耦合）高（纯展示 + prop 驱动）

React 推荐：事件处理器负责"响应交互"，Effect 负责"同步外部系统"。发现自己在多个地方写相同 DOM 操作时，收进 useEffect。
